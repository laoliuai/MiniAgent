# MiniAgent 功能演进路线图

> 日期：2026-03-13
> 状态：brainstorming 阶段，方案 C 已确认

---

## 项目背景

MiniAgent 是一个精简的 AI agent 框架，基于 MiniMax M2.5 模型，使用 Anthropic 兼容 API。当前已具备基本的 agent loop、工具系统、LLM 客户端、MCP 集成、技能系统和 ACP 服务端。

**主要使用场景**：个人使用，用来开发自用或企业内部的 agent（如类 OpenClaw 的生活助手平台）。目前不用于编程场景。

---

## 六大需求点

### 1. 工具层重构与增强
**现状**：已有 ReadTool（行寻址+offset/limit+token截断）、WriteTool（全量覆写）、EditTool（精确字符串替换）、BashTool（前台/后台/进程管理），缺少 GrepTool。
**目标**：优化现有工具质量（大文件保护、长行截断、批量替换），新增 GrepTool。
**用户痛点**：当前工具实现质量不够好，是最想优先改的部分。

### 2. 日志增强
**现状**：AgentLogger 已记录请求/响应/工具结果到文件，JSON 格式，但粒度不可配置，控制台无详细输出。
**目标**：分级日志（minimal/standard/verbose），可配置控制台和文件输出粒度，保证遇到问题时能看到每一次请求的 prompt 和 response。

### 3. 分层记忆与存储
**现状**：仅有 SessionNoteTool 将笔记写入 `.agent_memory.json`，无语义检索，无分层。
**目标**：实现本地轻量级分层记忆，但**接口抽象可插拔**，后续可无缝切换到 OpenViking、Mem0 等实现。
**调研发现**：
- **OpenViking**（火山开源）：三级上下文 L0/L1/L2，文件系统范式，内置记忆自迭代循环
- **Mem0**：自动从对话提取事实，优先级打分+衰减，26% 准确率提升
- **Letta/MemGPT**：LLM 自主管理"内存搬运"，核心记忆(RAM) + 归档记忆(Disk)
- **Zep/Graphiti**：时序知识图谱，双时间线模型

### 4. 上下文优化与压缩
**现状**：tiktoken 估算 token，超限时 LLM 总结整轮对话，保留所有 user 消息。工具结果全量保留在上下文中。
**目标**：工具结果智能裁剪（只保留必要上下文），选择性上下文加载，减少无效 token 消耗。
**调研发现**：
- Claude Code 三层压缩：微压缩（清旧工具输出）→ 自动压缩（总结对话）→ 手动压缩
- 上下文编辑（2025.9）：选择性清除过期工具调用，100轮评估减少 84% token 消耗
- Cursor：分层上下文 L1 当前文件 → L2 导入文件 → L3 相关文件 → L4 项目结构

### 5. Todo 功能
**现状**：无任务追踪能力。
**目标**：防止长链路任务跑偏，让 agent 能拆分任务、追踪进度、逐步执行。
**用户痛点**：与工具层并列为优先想做的功能。
**调研发现**：Claude Code 的 TodoWrite 用 content/status 模型，是 agent 自律+用户可见性的双重工具。

### 6. SubAgent
**现状**：无子 agent 能力。
**目标**：类 Claude Code 模式——主 agent 显式 spawn 子 agent，独立工具集和上下文，完成后返回结果。**需要支持并行执行**。
**调研发现**：
- Claude Code：Task tool spawn 子 agent，最多 10 个并发后台任务，子 agent 不能再 spawn
- OpenAI Swarm/Agents SDK：Agent + Handoff 两个原语，显式上下文传递
- CrewAI：角色驱动，自动分配管理者 agent

## 开发顺序：方案 C（双线并行，已确认）

```
第1阶段：工具层重构 + Grep + Todo + 日志增强
         （工具和日志改动互不依赖，可以一起做）

第2阶段：上下文优化
         （工具结果裁剪、选择性上下文、压缩策略升级）

第3阶段：SubAgent
         （并行执行，独立上下文，结果回传）

第4阶段：分层记忆
         （抽象接口 + 本地轻量实现，可插拔替换）
```

### 排序理由

1. **第1阶段优先**：用户明确工具是当前最大痛点，Todo 也急需。日志与工具正交可并行，且后续所有阶段调试都依赖好的日志。先把基本功补齐。

2. **第2阶段上下文优化**：必须在 SubAgent 之前完成。子 agent 会成倍增加上下文压力，如果没有好的裁剪策略，SubAgent 落地后体验会很差。

3. **第3阶段 SubAgent**：构建复杂 agent 的核心能力。有了优化过的上下文管理，子 agent 才能高效运行。并行执行是明确需求。

4. **第4阶段分层记忆放最后**：
   - 不是因为不重要，而是它最需要设计思考
   - 前三个阶段的实践（工具如何产出结果、上下文如何流转、子 agent 如何协作）会帮助验证记忆接口的抽象设计
   - 记忆模块最独立，不阻塞其他功能

## 调研关键发现

### 工具实现（行业共识）
- **所有主流框架都用行寻址**（Claude Code、Aider、SWE-agent、OpenCode），无人用字节偏移
- Claude Code Read：默认 2000 行，offset/limit 控制，超 token 阈值强制分页
- Claude Code Edit：精确字符串替换 + replace_all 选项
- Claude Code Grep：基于 ripgrep，三种输出模式（路径/内容/计数）
- SWE-agent 独特设计：edit 内置 linter 门控，语法错误的编辑直接拒绝
- Sweep AI：大文件先返回结构大纲（函数/类签名+行号），再按需读取，减少 90% token

### 记忆系统设计原则
- 接口抽象是关键：存储后端（本地文件/SQLite/向量DB/知识图谱）应可插拔
- 分层是共识：短期记忆（会话内）+ 长期记忆（跨会话）+ 归档（按需检索）
- OpenViking 的 L0/L1/L2 三级摘要模型适合作为本地实现的参考

### 上下文优化核心策略
- 工具输出是最大且价值衰减最快的 token 来源——优先清理
- 渐进式加载（progressive disclosure）：先给摘要，按需加载全量
- Claude Code 的"文件再水化"：压缩后重新读取活跃文件

---

## 设计原则

1. **接口抽象**：记忆、存储、LLM 客户端都通过抽象接口，实现可插拔
2. **渐进增强**：每个阶段独立可交付，不依赖后续阶段
3. **YAGNI**：只做当前需要的，不为假想需求过度设计
4. **与现有架构一致**：遵循当前的 async-first、Tool 基类、ToolResult 模式
